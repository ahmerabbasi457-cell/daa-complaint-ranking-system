Understanding the data structures and algorithms powering this system

🎯

Top-K Problem

The Top-K problem asks: given N elements, find the K largest (or highest-priority) elements efficiently — without sorting all N items.

Naïve (Sort All) O(N log N)
Min-Heap (K fixed) O(N log K)
Space Used O(K)

When K ≪ N, the heap approach is dramatically faster — ideal for real-time complaint queues.

📐

Score Formula

Score = (U × Wu + L × Wl) × e−λt
UUrgency weight (High=3, Med=2, Low=1)
WuUrgency importance factor
LNumber of likes / upvotes
WlLike importance factor
λDecay rate constant
tTime since submission (hours)
e−λtExponential time decay multiplier

Old complaints naturally lose score over time, ensuring fresh, relevant issues surface to the top.

⚔

Min-Heap vs Sorting

Min-Heap (Used) Full Sort
Time Complexity O(N log K) O(N log N)
Space O(K) O(N)
Live Updates ✔ Efficient ✘ Re-sort All
Insert Cost O(log K) O(N log N)
Memory for 1M items, K=10 ~10 slots ~1M slots
⚙

System Techniques

🏔
Min-Heap (Priority Queue)

Maintains the Top-K complaints. The root always holds the lowest-scored item, enabling O(log K) insertion and O(1) minimum access.

⏳
Exponential Time Decay

Score multiplied by e−λt. Recent complaints are boosted; stale complaints decay naturally — a greedy freshness heuristic.

🤖
Spam Detection (Clustering)

Duplicate and near-duplicate complaints are detected via text similarity clustering (e.g., Jaccard / cosine similarity), preventing spam inflation.

🔢
Greedy Top-K Selection

At each insert, greedily maintain only K elements by evicting the minimum-scored item — a classic greedy algorithm pattern.
